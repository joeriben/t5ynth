# T5ynth IPC Protocol

This document is the authoritative specification of the binary pipe protocol
between the JUCE plugin (`src/inference/PipeInference.cpp`, referred to as
"client") and the Python inference server (`backend/pipe_inference.py`,
referred to as "server"). It is written so that a developer replacing the
Python side with a different implementation can do so without reading the
C++ client.

All integers and floats are **little-endian**. No byte-order marker or magic
is present; both sides are hard-coded to little-endian layout via Python's
`struct` format strings (`'<iiiiif'`, `'<H'`, `'<I'`) and the C++ side's
raw `memcpy` into packed structs on the only supported host architectures
(x86_64 and arm64, both little-endian). A replacement server MUST use
little-endian encoding.

---

## 1. Why a subprocess

Inference runs in a Python subprocess for two reasons:

1. **`torchsde.BrownianTree` has no clean C++ equivalent.** The Stable Audio
   Open pipeline uses a DPM++ SDE sampler whose Brownian tree noise is
   generated by `torchsde` (Python-only). Porting the sampler to C++ would
   mean rewriting `torchsde._brownian.brownian_interval` and the associated
   scheduler, and would break determinism across platforms. See
   `backend/pipe_inference.py:26-49` for the deterministic noise patch.
2. **Python is the canonical host for `diffusers`, `transformers`, and
   `stable_audio_tools`.** Reimplementing any of these in C++ is out of
   scope.

The subprocess uses a binary pipe over `stdin`/`stdout` instead of HTTP for
three reasons:

1. No HTTP server setup, no port allocation, no firewall issues.
2. Lower latency: no TCP handshake per request, no JSON response body, raw
   float32 PCM is written directly to the parent's read end.
3. The subprocess stays alive across generations — pipelines are loaded
   once, not per request.

## 2. Subprocess lifecycle

### 2.1 Executable resolution

`PipeInference::launch(backendDir)` resolves the executable in this order
(`src/inference/PipeInference.cpp:15-33`):

1. **Bundled PyInstaller binary** — `backendDir/dist/pipe_inference/pipe_inference`
   (dev layout) or `backendDir/pipe_inference[.exe]` (installed layout next to
   the backend dir). If found, no script path is passed; the frozen binary
   executes itself.
2. **Python script fallback** — `backendDir/pipe_inference.py`, launched via
   a Python interpreter. The interpreter is searched in this order
   (`src/inference/PipeInference.cpp:35-59`):
   - `<projectRoot>/.venv/bin/python` (and `venv/`, `python3` variants)
   - On POSIX only: `/usr/bin/python3`, `/usr/local/bin/python3`
   - Fallback: the literal string `"python3"` on `PATH`.

If neither exists, `launch()` returns `false` and logs
`"pipe_inference.py not found"`.

### 2.2 Environment and working directory

No environment variables are explicitly set or cleared by the client. On
POSIX the child is produced by `fork()` + `execlp()` (so it inherits the
parent's full environment and cwd,
`src/inference/PipeInference.cpp:220-238`). On Windows the child is produced
by `CreateProcessW()` with `lpEnvironment = nullptr` (inherit parent env)
and `lpCurrentDirectory = nullptr` (inherit parent cwd,
`src/inference/PipeInference.cpp:181-182`).

If a replacement server needs a specific cwd (e.g. for relative model
paths), it MUST resolve paths from absolute locations internally. The
server currently uses absolute platform paths for model discovery
(`backend/pipe_inference.py:76-92`: `~/Library/T5ynth/models`,
`~/.local/share/T5ynth/models`, `~/t5ynth/models`).

### 2.3 Pipe plumbing

- **POSIX**: two anonymous pipes. `pipeIn[1]` (parent write) is dup'd to
  child stdin; `pipeOut[0]` (parent read) receives child stdout. Child
  stderr is inherited and goes to the parent's terminal (when launched from
  a terminal) or is dropped (when launched from a bundled `.app`).
  Code: `src/inference/PipeInference.cpp:203-246`.
- **Windows**: `CreatePipe` × 2 with `bInheritHandle=TRUE`, then
  `SetHandleInformation(..., HANDLE_FLAG_INHERIT, 0)` on the parent-side
  handles so they are NOT inherited. `si.hStdError = STD_ERROR_HANDLE`
  (child stderr = parent stderr).
  Code: `src/inference/PipeInference.cpp:139-200`.

The child's stdin is line-buffered (Python `for line in sys.stdin:` at
`backend/pipe_inference.py:960`). The child's stdout is explicitly written
as raw bytes via `sys.stdout.buffer.write(...)` followed by
`sys.stdout.buffer.flush()` (see `send_ready`, `send_audio`, `send_error`
at `backend/pipe_inference.py:882-926`). Text writes from imported packages
would corrupt the pipe; the server redirects `sys.stdout = sys.stderr`
around any code path known to print (native pipeline loading at
`backend/pipe_inference.py:269-280` and generation at
`backend/pipe_inference.py:707-720`).

### 2.4 Handshake ("ready" signal)

After launch, the server loads the default model on the first working
device (best-first: `mps` → `cuda` → `cpu`) and sends a **single "ready"
frame** on stdout:

```
\x02 <uint16_le length> <UTF-8 JSON bytes>
```

Written by `send_ready()` at `backend/pipe_inference.py:882-893`, parsed by
`launch()` at `src/inference/PipeInference.cpp:248-294`.

The JSON object has this shape:

```json
{
  "devices": ["mps", "cpu"],
  "default": "mps",
  "models": ["stable-audio-open-1.0", "audioldm2"],
  "default_model": "stable-audio-open-1.0"
}
```

- `devices` — the list of device backends the UI may request
  (`available_devices()`: `mps`/`cuda` when present, plus `cpu`).
- `default` — the device on which the default model was successfully
  preloaded during startup.
- `models` — all discovered model directory names.
- `default_model` — `"stable-audio-open-1.0"` if present, else the first
  discovered model.

The client times out after **120 seconds** waiting for the `\x02` byte
(single-pipeline warm load can still take tens of seconds,
`src/inference/PipeInference.cpp:248-257`), then 5 seconds for the payload.

If the server cannot load any pipeline, it sends an error frame
(`\x00 + uint32 + utf8`) via `send_error()` and exits
(`backend/pipe_inference.py:936-953`). The client does not currently parse
an error frame during handshake — it will fail on the unexpected first
byte and log `"unexpected ready byte"`.

### 2.5 Keep-alive

The subprocess stays alive between generations. There is no heartbeat. The
server's main loop blocks on `sys.stdin` (`backend/pipe_inference.py:960`)
and processes one request per newline-terminated JSON object.

### 2.6 Shutdown

`PipeInference::shutdown()` (`src/inference/PipeInference.cpp:302-333`)
closes both pipe ends and then:

- **POSIX**: sends `SIGTERM`, waits up to 3 s (30 × 100 ms `waitpid` polls),
  then sends `SIGKILL` and reaps.
- **Windows**: `TerminateProcess(..., 0)` with a 3 s `WaitForSingleObject`.

Closing stdin causes the server's `for line in sys.stdin:` loop to fall
through to EOF, which exits `main()` cleanly. `SIGTERM`/`TerminateProcess`
is a fallback if the server is stuck inside inference.

### 2.7 Restart-on-crash

`PipeInference::generate()` calls `isChildAlive()` (via `waitpid(..., WNOHANG)`
on POSIX or `GetExitCodeProcess` on Windows,
`src/inference/PipeInference.cpp:70-83`) before and after writing. If the
child has died, `tryRestart()` → `shutdown()` + `launch(backendDir_)` is
called. Two restart scenarios:

- **Pre-write check** (`src/inference/PipeInference.cpp:410-419`): if the
  child is dead, the restart happens transparently and the request
  continues on the fresh subprocess.
- **Write or read failure** (`src/inference/PipeInference.cpp:473-498`):
  the restart happens, but the current request is aborted and the client
  surfaces `"Inference restarted — try again"` or
  `"Inference crashed — restarted, try again"` to the UI. The caller is
  expected to re-issue the request.

The launch path is guarded by a `std::atomic<bool> launching_` to prevent
concurrent re-launches (`src/inference/PipeInference.cpp:97-102`).

## 3. Request wire format

Each request is a **single-line UTF-8 JSON object** written to the child's
stdin, terminated by `\n`. The client explicitly strips embedded `\n` and
`\r` from the serialized JSON before writing
(`src/inference/PipeInference.cpp:468-469`):

```cpp
auto jsonStr = juce::JSON::toString(juce::var(json.get()), true);
jsonStr = jsonStr.removeCharacters("\n\r") + "\n";
```

There is no size prefix and no acknowledgment of request receipt. The
server reads one line and dispatches it.

### 3.1 Field schema

All fields below are from `PipeInference::generate()` serialization
(`src/inference/PipeInference.cpp:427-470`) and
`backend/pipe_inference.py:735-877` (deserialization).

| Field              | JSON type                 | Units        | Required | Default (server)                   | Notes                                                                                                                                         |
| ------------------ | ------------------------- | ------------ | -------- | ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| `prompt_a`         | string                    | —            | yes      | `""`                               | Primary text prompt.                                                                                                                          |
| `prompt_b`         | string                    | —            | no       | (absent) → single-prompt path      | Only serialized if non-empty (`PipeInference.cpp:430-431`).                                                                                    |
| `alpha`            | number                    | unitless     | no       | `0.0`                              | Interpolation weight between A and B. `0` = midpoint, `-1` = pure A, `+1` = pure B. Values outside `[-1, 1]` renormalize toward the nearer pole. |
| `magnitude`        | number                    | unitless     | no       | `1.0`                              | Scales the manipulated embedding. Skipped if `abs(magnitude - 1) < 1e-6`.                                                                       |
| `noise_sigma`      | number                    | unitless     | no       | `0.0`                              | Gaussian noise stddev added to the embedding. Noise is drawn from `numpy.random.Generator(PCG64(seed))` for cross-platform determinism.        |
| `duration`         | number                    | seconds      | no       | `3.0`                              | Output audio length. SA Open path truncates to `ceil(duration * 44100)` samples.                                                              |
| `start_pos`        | number                    | 0.0–1.0      | no       | `0.0`                              | SA Open only: maps to `audio_start_in_s` / `audio_end_in_s` via a `virtual_total = duration / (1 - start_pos)` trick (`pipe_inference.py:767-769`). AudioLDM2 and native paths **ignore** this field. |
| `steps`            | integer                   | —            | no       | `20` (diffusers), `50` (AudioLDM2), `8` (native) | Number of diffusion inference steps.                                                                                        |
| `cfg_scale`        | number                    | —            | no       | `7.0` (diffusers), `3.5` (AudioLDM2), `4.0` (native) | Classifier-free guidance scale.                                                                                              |
| `seed`             | integer                   | —            | no       | `-1` → random                       | `-1` means server picks `random.randint(0, 2**31 - 1)`. The chosen seed is echoed in the response header.                                      |
| `device`           | string                    | —            | no       | `default_device` from handshake    | One of `"mps"`, `"cuda"`, `"cpu"`, or `"auto"`. Unknown/unreachable values fall through to `default_device`.                                    |
| `model`            | string                    | —            | no       | `default_model` from handshake     | Model directory name. Unknown values fall through to `default_model`.                                                                          |
| `mode`             | string                    | —            | no       | `"generate"`                       | One of `"generate"`, `"preload"`, `"interpolate"`, `"decode_cached"`. The latter two are latent-cache operations; see §3.3.                     |
| `dimension_offsets`| array of `[int, number]`  | unitless     | no       | (absent) → no-op                   | DimensionExplorer offsets. Each pair is `[dim_index, delta]`. Applied as `manipulated[:, :, idx] += delta` (`pipe_inference.py:815-819`). `dim_index` out of range is silently ignored. This is a **sparse** list, not a 768-element dense array. |
| `semantic_axes`    | object `{string → number}`| unitless     | no       | (absent) → no-op                   | Per-axis deltas. Keys MUST match the server's `SEMANTIC_AXIS_POLES` dict (`pipe_inference.py:380-389`): `music_noise`, `acoustic_electronic`, `improvised_composed`, `refined_raw`, `solo_ensemble`, `sacred_secular`, `tonal_noisy`, `rhythmic_sustained`. Unknown keys are silently dropped. `abs(value) < 0.001` is skipped. |
| `latent_a`         | string                    | —            | mode=interpolate | —                            | Name of a previously cached latent (SA Open only).                                                                                             |
| `latent_b`         | string                    | —            | mode=interpolate | —                            | Name of a previously cached latent.                                                                                                            |
| `lerp_alpha`       | number                    | 0.0–1.0      | mode=interpolate | `0.5`                        | Linear interpolation weight.                                                                                                                   |
| `cache_as`         | string                    | —            | no       | (absent)                            | SA Open only. If set, the pre-VAE latent is cached under this name for later `interpolate` / `decode_cached`. The response still carries the decoded audio. |
| `latent_name`      | string                    | —            | mode=decode_cached | —                          | Name of the cached latent to decode.                                                                                                           |

### 3.2 Escaping

The request is serialized by JUCE's `juce::JSON::toString(..., true)`
("compact" form) and then has all `\n` and `\r` stripped
(`src/inference/PipeInference.cpp:468-469`). A final `\n` is appended as
the line terminator. Embedded newlines in prompt text would corrupt the
framing, so they are removed before transmission. **Tabs and other
control characters are NOT stripped**; a replacement client that allows
them in prompts should be aware that JSON escapes (`\t`, `\"`, etc.) are
valid.

### 3.3 Special modes

- **`"preload"`** (`backend/pipe_inference.py:977-985`): loads the
  `(model, device)` pipeline into the cache and responds with a zero-sample
  audio frame (same header layout, `samples=0`, no PCM body). Used by
  `PipeInference::preload()` to warm the pipeline on device switch. The
  client reads and discards the response (`PipeInference.cpp:562-613`).
- **`"interpolate"`** (`backend/pipe_inference.py:990-993`): blends two
  cached latents and VAE-decodes. SA Open only — raises on AudioLDM2.
- **`"decode_cached"`** (`backend/pipe_inference.py:994-997`): VAE-decodes
  a single cached latent. SA Open only.

`interpolate` and `decode_cached` are not currently called from the JUCE
client based on a grep of `src/` — their wire format is defined server-side
only.

## 4. Response wire format

Each response starts with a single **status byte**:

| Byte     | Meaning                           |
| -------- | --------------------------------- |
| `\x00`   | Error (see §4.3)                  |
| `\x01`   | Audio (see §4.1)                  |
| `\x02`   | Ready (only once, during handshake — never sent in response to a request) |

Any other value causes the client to abort the current request with
`"Unexpected response: <n>"` (`src/inference/PipeInference.cpp:514-518`).

### 4.1 Success frame (`\x01`)

Layout:

```
\x01
<header : 24 bytes>
<pcm    : channels * samples * 4 bytes, planar float32 little-endian>
<emb    : 2 + 4*num_dims + 4*num_dims bytes>
```

The header is packed by Python as `struct.pack('<iiiiif', ...)`
(`backend/pipe_inference.py:904`) and parsed by C++ as a raw memcpy into
this struct (`src/inference/PipeInference.cpp:521`):

```cpp
struct { int32_t flag, samples, channels, sampleRate, seed; float timeMs; } header;
```

| Offset | Width | Type     | Field        | Semantics                                                                             |
| ------ | ----- | -------- | ------------ | ------------------------------------------------------------------------------------- |
| 0      | 4     | int32 LE | `flag`       | Always `1` on success (`pipe_inference.py:904`). The client does not inspect it.      |
| 4      | 4     | int32 LE | `samples`    | Per-channel sample count. `0` for preload responses.                                  |
| 8      | 4     | int32 LE | `channels`   | Almost always `2`. SA Open VAE outputs stereo; AudioLDM2 mono is duplicated to stereo (`pipe_inference.py:603-604`). |
| 12     | 4     | int32 LE | `sampleRate` | Hz. Currently always `44100`.                                                          |
| 16     | 4     | int32 LE | `seed`       | The actual RNG seed used (resolved from `-1` or passed through).                      |
| 20     | 4     | float32 LE | `timeMs`   | Generation time. **Note**: Python packs `elapsed_ms * 1000` (`pipe_inference.py:904`) where `elapsed` is already in seconds. The C++ side then divides by 1000 in the UI (`PromptPanel.cpp:675`: `result.generationTimeMs / 1000.0f`). So the wire value is microseconds, but the field name on both sides is `timeMs`. This is a **known naming inconsistency** that a replacement server MUST preserve bit-for-bit. |

**Note on `flag`**: despite being named "flag", it is not an error flag.
Errors use the out-of-band `\x00` status byte. A replacement server should
set `flag = 1` for forward compatibility but can leave it unused.

The PCM body is `channels * samples` float32 values, **planar** (channel 0
in full, then channel 1 in full, etc. — NOT interleaved). This is confirmed
by:

- Python: `pcm = audio_np.astype(np.float32).tobytes()` where `audio_np`
  has shape `[channels, samples]` (`pipe_inference.py:902-903`). NumPy's
  default C order produces channel-major bytes.
- C++: the client reads the whole blob into `std::vector<float> pcm` and
  then copies channel `ch` from offset `ch * samples`
  (`src/inference/PipeInference.cpp:538-543`).

### 4.2 Embedding footer

After the PCM body, the server appends:

```
<uint16 LE num_dims>
<float32[num_dims] emb_a>    // only if num_dims > 0
<float32[num_dims] emb_b>    // only if num_dims > 0
```

Written by `send_audio()` at `backend/pipe_inference.py:908-916`, parsed by
`PipeInference::generate()` at
`src/inference/PipeInference.cpp:549-557`.

- `num_dims` is `768` in the diffusers (SA Open) and native paths (mean-pooled
  T5 encoder output). `emb_b` is a zero vector when `prompt_b` is empty
  (`pipe_inference.py:801`).
- `num_dims` is `0` for the AudioLDM2 path (`_generate_audioldm2` returns
  `emb_stats=None`, `pipe_inference.py:607`; `send_audio`'s else branch
  writes `struct.pack('<H', 0)`, line 915).
- `num_dims` is `0` for preload responses (`send_audio(empty, 44100, 0, 0.0)`
  is called without `emb_stats`, `pipe_inference.py:984`).

The client reads the `num_dims` field with a 5-second timeout and populates
`Result::embeddingA` / `Result::embeddingB` if non-zero
(`src/inference/PipeInference.cpp:549-557`).

A replacement server MUST emit the `uint16` count even when it has no
embedding data — omitting it would desynchronize the pipe for the next
request.

### 4.3 Error frame (`\x00`)

Layout:

```
\x00 <uint32 LE length> <UTF-8 message bytes>
```

Python: `send_error()` at `backend/pipe_inference.py:919-925`:

```python
sys.stdout.buffer.write(b'\x00')
sys.stdout.buffer.write(struct.pack('<I', len(msg_bytes)))
sys.stdout.buffer.write(msg_bytes)
```

C++: `src/inference/PipeInference.cpp:500-512`:

```cpp
if (status == '\x00')
{
    juce::uint32 msgLen = 0;
    if (readExact(&msgLen, 4))
    {
        std::vector<char> msg(msgLen + 1, 0);
        readExact(msg.data(), static_cast<int>(msgLen));
        result.errorMessage = juce::String::fromUTF8(msg.data(), static_cast<int>(msgLen));
    }
    ...
}
```

The error frame is **in-band** on stdout. The server catches any exception
during request dispatch (`backend/pipe_inference.py:1002-1007`), prints a
traceback to stderr, and sends a `\x00` frame with `str(e)`. The subprocess
continues running after sending an error — the next request is handled
normally.

### 4.4 Out-of-band logging (stderr)

All Python `logging` output is configured at
`backend/pipe_inference.py:51-52` to write to `sys.stderr` with the format
`"%(asctime)s [%(name)s] %(message)s"`. Stderr is inherited by the parent
process. Typical lines:

- `"... [pipe_inference] Loading diffusers pipeline from /.../stable-audio-open-1.0 on mps..."`
- `"... [pipe_inference] Generating on mps: '<prompt>' (3.0s, 20 steps, CFG=7.0, seed=12345)"`
- `"... [pipe_inference] Generated in 4.7s"`
- `"... [pipe_inference] Request failed (model=..., mode=generate): <exception>"`

Full Python tracebacks on failure are printed via
`traceback.print_exc(file=sys.stderr)` (`pipe_inference.py:1005-1006`).
Stderr is the only place to see them — the in-band error frame carries
only `str(e)`.

## 5. Determinism guarantees

Same `seed` + same `prompt_a` / `prompt_b` + same manipulation params + same
`model` MUST produce **bit-identical** audio across CPU, CUDA, ARM, and x86.
This is load-bearing for preset reproducibility.

The guarantee is achieved by:

1. **Monkey-patching `torchsde._brownian.brownian_interval._randn`** to use
   `numpy.random.Generator(PCG64(seed))` instead of `torch.randn` on the
   target device. `torch.Generator("cpu")` and `torch.Generator("cuda")`
   are different PRNGs (same seed → different sequences); numpy's PCG64
   is identical on all platforms.
   See `backend/pipe_inference.py:26-49`.
2. **All noise injection** (embedding noise at `pipe_inference.py:550-553`
   and `809-812`, also `669-673`) uses `np.random.Generator(np.random.PCG64(seed))`
   directly, never `torch.randn`.
3. **Brownian noise inside the SDE sampler** is replaced transparently via
   the monkey-patch above.

A replacement server MUST reproduce this behaviour: any use of
`torch.randn`, `torch.Generator(device)`, or platform-dependent RNGs in
the noise path will break preset determinism.

## 6. Threading and backpressure

**The pipe is strictly synchronous.** One request → one response. There is
no request pipelining, no request ID, and no cancellation.

- The server's main loop (`backend/pipe_inference.py:960-1007`) processes
  requests one line at a time, in order.
- `PipeInference::generate()` is documented as "Blocking — call from
  background thread" (`src/inference/PipeInference.h:74-76`) and takes no
  internal lock.
- There is **no mutex around the pipe**. The client relies on higher-level
  guards to prevent concurrent calls. The UI uses a per-panel `generating`
  boolean flag (`src/gui/PromptPanel.cpp:625-627, 651-652, 704-708`) to
  debounce multiple triggers.
- If two threads were to call `generate()` concurrently, their writes to
  stdin could interleave arbitrary bytes, corrupting the JSON framing. A
  replacement client that supports concurrent callers MUST add its own
  mutex or queue around `writeExact` / `readExact`.

### 6.1 Timeouts

- Handshake `\x02`: 120 000 ms (`PipeInference.cpp:251`).
- Handshake JSON payload: 5 000 ms (`PipeInference.cpp:269, 272`).
- Response status byte: 180 000 ms (3 min — allows for dual-pipeline
  startup on first call, `PipeInference.cpp:486`).
- Response header, PCM, and embeddings: 120 000 ms default in `readExact`
  (`PipeInference.h:109`), explicitly 5 000 ms for the embedding footer
  (`PipeInference.cpp:551, 555, 556`).

On any timeout, the client calls `isChildAlive()` and either restarts the
subprocess (and surfaces `"Inference crashed — restarted, try again"`) or
surfaces `"Timeout waiting for response"`.

## 7. Error propagation

### 7.1 In-band (`\x00`)

See §4.3. Any exception raised during request handling is caught at
`backend/pipe_inference.py:1002-1007`, logged with a traceback to stderr,
and sent as a `\x00` frame with `str(e)` as the UTF-8 payload. The server
continues.

### 7.2 Out-of-band (stderr)

Log lines and tracebacks as described in §4.4. The client does NOT read
stderr — it only surfaces the in-band `\x00` message.

### 7.3 Pipe-level errors

- Write failure (`writeExact` returns `false`) → treated as subprocess
  death; `tryRestart()` is called; caller sees
  `"Inference restarted — try again"` or
  `"Inference crashed — restart failed"` (`PipeInference.cpp:473-482`).
- Read failure on the status byte → same handling
  (`PipeInference.cpp:486-498`).
- Read failure on header / PCM / embeddings → the current request fails
  with a specific message (`"Failed to read header"`,
  `"Failed to read PCM data (N bytes)"`); the subprocess is NOT restarted.
  The next request will likely fail on a stale pipe; a restart occurs only
  on the next generate() call's `isChildAlive` check.

### 7.4 UI surfacing

`Result::errorMessage` is forwarded verbatim to the status bar via the
`onStatusChanged` callback (`src/gui/PromptPanel.cpp:689` and `:754`).
There is no dedicated error UI — the generate button is re-enabled and
the message is shown next to it.

## 8. Example round-trip

### 8.1 Request

A typical single-prompt request from the UI:

```json
{"prompt_a":"a glass bell ringing","alpha":0.0,"magnitude":1.0,"noise_sigma":0.0,"duration":3.0,"start_pos":0.0,"steps":20,"cfg_scale":7.0,"seed":42,"device":"mps","model":"stable-audio-open-1.0"}
```

Followed by `\n`. Note the compact JSON with no whitespace between tokens
(JUCE `juce::JSON::toString(var, true)`).

### 8.2 Success response header (hex dump)

For `samples=132300`, `channels=2`, `sampleRate=44100`, `seed=42`,
`elapsed=4.723` s, the 25 bytes on the wire are:

```
01                                       status byte \x01 (success)
01 00 00 00                              flag = 1
0c 04 02 00                              samples = 132300
02 00 00 00                              channels = 2
44 ac 00 00                              sampleRate = 44100
2a 00 00 00                              seed = 42
00 a0 93 46                              timeMs ≈ 4723.0 (float32 LE,
                                          which is elapsed_ms * 1000,
                                          i.e. microseconds — see §4.1)
```

Followed by `2 * 132300 * 4 = 1 058 400` bytes of planar float32 PCM, then
`03 00` (uint16 `num_dims = 768`), then `4 * 768 = 3072` bytes of
`emb_a`, then `3072` bytes of `emb_b`. Total response size:
`1 + 24 + 1 058 400 + 2 + 3072 + 3072 = 1 064 571` bytes.

## 9. Versioning

**There is no protocol version field.** Neither the handshake nor the
response header carries a version number. The implicit version is
determined by the commit hash of both sides.

Consequences:

- Any protocol change (adding a field, changing the header layout,
  changing the embedding footer presence rules) MUST be atomic across both
  sides. There is no negotiation.
- Adding a new optional JSON request field is backward-compatible if the
  server uses `request.get(key, default)` (the current pattern) — old
  clients simply won't send the field.
- Adding to the response side is NOT backward-compatible: an old client
  would read the "new" bytes as the start of the next response, corrupting
  the pipe on the following request.
- The `flag` field in the response header (currently always `1`) is a
  candidate location for a future version byte but is not used as one
  today.

A replacement server MUST match the exact byte layout described in §4 or
the C++ client will misinterpret the stream.

## 10. Known constraints and caveats

1. **`timeMs` is actually microseconds.** See §4.1. `struct.pack('<iiiiif', 1, samples, channels, sr, seed, elapsed_ms * 1000)` at `backend/pipe_inference.py:904` where `elapsed_ms` is in fact `elapsed` in seconds (assigned at `backend/pipe_inference.py:873, 595, 722`, then passed as `elapsed` from `main()` at line 1001). The C++ side compensates by dividing `result.generationTimeMs / 1000.0f` in the UI layer (`PromptPanel.cpp:675`). **Not verified — reader should re-check the unit math end-to-end.** The invariant a replacement server must preserve is: the wire float32 value, divided by 1000, yields seconds.

2. **PCM is planar, not interleaved.** See §4.1. A replacement client that
   feeds the bytes directly to an interleaved-only audio API will produce
   garbled output.

3. **`start_pos` only affects Stable Audio Open.** The AudioLDM2 and native
   paths ignore it entirely. Preset recall on a switched model will
   silently change behaviour.

4. **`dimension_offsets` is sparse, not dense.** It is serialized as an
   array of `[index, value]` pairs (`PipeInference.cpp:446-457`), not a
   768-element dense array. An index-out-of-range value is silently dropped
   server-side (`pipe_inference.py:558-560, 678-680, 817-819`).

5. **`semantic_axes` keys are enum-constrained.** Only the eight keys in
   `SEMANTIC_AXIS_POLES` (`pipe_inference.py:380-389`) are honored. Typos
   are silently dropped.

6. **AudioLDM2 has a known generation failure inside the bundle**
   ("The size of tensor a (N) must match the size of tensor b (M) at
   non-singleton dimension 1"). This is a `diffusers` / pipeline-state
   issue, not a pipe protocol issue. See `HANDOVER_AUDIOLDM2.md` for
   context. A replacement server can ignore this if it implements its
   own AudioLDM2 path.

7. **The server redirects `sys.stdout = sys.stderr` in two places** to
   protect the pipe from libraries that print to stdout
   (`pipe_inference.py:269-280` for native model loading,
   `pipe_inference.py:707-720` for native generation). A replacement
   server must similarly suppress all non-protocol stdout writes.

8. **PyInstaller runtime hook disables `multiprocessing.resource_tracker`.**
   The frozen binary is `sys.executable`, so the tracker would re-exec the
   full app (fork bomb). See `backend/runtime_hook.py:14-23`. A replacement
   server built with PyInstaller MUST replicate this or avoid any code
   path that starts `multiprocessing`.

9. **CUDA libraries are stripped from the bundle** to save ~800 MB:
   `nccl`, `cufft`, `cusparse`, `cusolver`, `nvrtc`, `nvJitLink`, `cupti`,
   `triton` are dropped in `backend/pipe_inference.spec:106-120`. A
   replacement server using any of these libraries at inference time will
   fail inside the bundle.

10. **Preset determinism depends on the torchsde monkey-patch.** Any
    replacement server that ships a different SDE sampler will not
    reproduce existing SA Open presets bit-for-bit. If that matters, the
    replacement must reimplement the same numpy PCG64 noise ordering.
